# 노트플러스P 모바일 대용량 복원 보호 설계 v1

작성일: 2026-07-23  
대상 후보: `노트앱_v16.html`

## 문제

- 실제 Evernote 이전 데이터는 첨부가 2,250개이므로 휴대폰 저장공간보다 클 수 있다.
- v15는 복원 실패 시 롤백하지만, 저장공간 부족을 다운로드 전에 차단하지 않아 시간·데이터 사용량을 낭비할 수 있다.

## 설계

1. manifest 검증 뒤 첨부 총 바이트를 계산한다.
2. `navigator.storage.estimate()`가 있으면 남은 공간과 필요 공간을 비교한다.
3. 필요 공간은 첨부 총량 + 상태/IndexedDB 여유분으로 잡는다. 여유분은 `max(64MB, 첨부의 15%)`다.
4. 공간이 부족하면 어떤 첨부도 다운로드하거나 쓰지 않고 차단한다.
5. 저장공간 API가 없으면 이를 정직하게 알리고 사용자의 명시 확인 뒤 진행할 수 있다.
6. 데이터 절약 모드·2G·3G에서는 예상 전송량을 경고하되 자동 차단하지 않는다.
7. 복원 중에는 처리 첨부 수와 누적 바이트를 표시한다.
8. 복원 직전 저장공간을 다시 검사해 미리보기 뒤 공간 변화도 잡는다.

## 무손실 관문

- 사전점검 차단 전후 상태 서명이 같아야 한다.
- 차단 시 `downloadAttachment` 호출 수는 0이어야 한다.
- 기존 UID·SHA·충돌 재매핑·복원 전 자동 보관·롤백은 그대로 유지한다.
- 저장공간 검증 자체가 실패해도 앱 전체는 멈추지 않고 검증 불가 상태를 고지한다.
