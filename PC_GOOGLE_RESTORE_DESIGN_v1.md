# 노트플러스P PC Google 무손실 복원 설계 v1

기준일: 2026-07-22

## 범위

- 같은 Google UID의 `current.json`이 가리키는 마지막 완성 백업만 읽는다.
- 첫 클릭은 pointer·manifest·수량·첨부 참조를 읽기 전용으로 검증하며 로컬 상태를 바꾸지 않는다.
- 두 번째 명시적 확인 뒤에만 첨부를 내려받고 상태를 복원한다.
- 자동 동기화·병합·공동 편집은 포함하지 않는다.

## 무손실 순서

1. 로그인 UID와 cloud 경로 UID가 같은지 검사한다.
2. `current.json`과 manifest 크기·SHA-256·형식·snapshot id를 검사한다.
3. 노트 수·첨부 수·노트별 첨부 소유관계를 검사한다.
4. 복원 미리보기 수량을 표시한다. 이 단계까지 로컬 상태는 불변이다.
5. 각 첨부를 순차 다운로드하고 서버 메타 SHA와 실제 Blob SHA를 모두 검사한다.
6. 로컬과 같은 attachment id인데 내용이 다르면 새 id로 재매핑해 기존 Blob을 보존한다.
7. 모든 첨부 검증 뒤 현재 로컬 상태를 `Google 복원 전 자동 보관` 복구 지점으로 새 상태 안에 포함한다.
8. 상태를 마지막에 한 번 커밋하고 다시 읽어 서명을 비교한다.
9. 실패하면 이전 상태를 재기록·검증하고 이번 시도의 새 첨부만 제거한다.
10. 복구 지점이 참조하는 첨부는 고아 Blob 정리에서도 삭제하지 않는다.

## 검증 완료 범위

- 미리보기 무변경
- 계정 전환 차단
- 다운로드 실패 무손실
- attachment id 충돌 재매핑
- 복원 전 로컬 상태·첨부 보존
- 고아 Blob 정리 뒤 복구 지점 첨부 보존
- 재접속 뒤 복원 노트·Blob 유지

실제 Firebase 계정과 실제 사용자 백업은 외부 프로젝트 설정 뒤 별도 검증한다.
