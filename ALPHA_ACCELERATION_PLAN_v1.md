## 1. 재점검 결론

기존 우선순위는 필요한 일을 폭넓게 담았지만, 알파를 가장 빨리 끝내는 순서는 아니었다.

- 사용자 의존 작업이 목록 앞을 막으면 Codex가 수행할 수 있는 후속 검증까지 대기 상태가 됐다.
- iPhone 교차 확인, 10,000노트 벤치마크, 전체 브라우저 매트릭스, 노트북 매핑, 태그 정규화 같은 항목은 현재 핵심 여정의 P0/P1이 없는 한 제한 알파 종료 조건이 아니다.
- 실제 ENEX 미리보기는 통과했지만 최종 저장·재접속 증거가 없어, 기능 추가보다 실제 이전 확정이 우선이다.
- 파일럿 모집을 기술 작업 뒤에 직렬 배치하면 가장 긴 외부 대기시간이 마지막에 발생한다. 모집은 지금 바로 병렬 시작해야 한다.

따라서 앞으로는 **최대 작업량**이 아니라 **최소 달력시간**을 최적화한다. 임계 경로는 한 줄로 유지하고, 사용자·기기·참가자 대기시간에는 별도의 자율 작업열을 계속 소진한다.

## 2. 알파 범위와 5개 관문

| 관문 | 상태 | 현재 근거 | 남은 최소 증거 |
|---|---|---|---|
| 누구나 여는 공개 릴리스 | 통과 | 2026-07-22 공개 루트·v12·SW HTTP 200, 공개/로컬 SHA 일치 | RC 재확인 |
| 실제 모바일 핵심 여정 | 부분 | 과거 v8 Samsung 통과, v12 자동 모바일 통과 | 공개 v12 실제 Android 1대 |
| 저장·데이터 안전 | 통과 | schema 4→5, IDB/보조저장, 취소·재시도, 766MB 무변경 미리보기 | 실제 최종 가져오기 결과 기록 |
| 비개발자 파일럿 3~5명 | 미통과 | 과업 카드·샘플 준비 | 최소 3명 결과 |
| 종료 품질 | 부분 | 현재 자동·공개 범위 P0/P1=0 | 실기기·파일럿 뒤 P0=0, 잔여 한계 보고 |

공식 진척도는 **통과 2/5, 부분 2, 미통과 1 = 40%**다. 구현량·자동검사 수를 이 수치에 섞지 않는다.

## 3. 최대 처리량 임계 경로

```text
공개 v12 고정
  → 실제 ENEX 최종 가져오기와 1,950/2,272 대조
  → v12 실제 Android 찾기·수정·재접속
  → 비개발자 파일럿 3명 병렬 실행
  → 관찰된 P0 즉시 / 재현 P1만 최소 수정
  → 공개 RC·종료 보고
```

### 사용자 개입을 두 번으로 묶는 실행 창

1. **사용자 창 A, 20~40분**
   - 공개 v12에서 실제 ENEX 선택
   - 미리보기 `노트 1,950개·첨부 2,272개·오류 0개` 확인
   - `선택한 노트 가져오기` 확정
   - 완료 화면을 유지하고 휴대폰 무선 디버깅 연결
   - 이 한 번의 창 뒤 Codex가 수량·첨부·저장·Android 검증을 연속 수행할 수 있게 한다.
2. **사용자 창 B, 참가자 조율만**
   - 비개발자 최소 3명에게 기존 10분 과업 카드를 동시에 전달
   - 참가자는 개인 ENEX가 아니라 제공된 샘플을 사용
   - 세 결과를 한 번에 회수해 수정 왕복 횟수를 한 차례로 줄인다.

파일 선택과 최종 가져오기 확인, 실제 기기 연결, 실제 참가자 행동은 브라우저·개인정보·물리 환경 경계 때문에 Codex가 증거를 만들어 대신할 수 없다.

## 4. 가장 빠른 개발 스케줄

| 구간 | 경과 목표 | 주체 | 작업 | 다음 구간 조건 |
|---|---:|---|---|---|
| A0 자율 사전준비 | 지금~4시간 | Codex | 대조표·무결성 검사·RC/파일럿 판정 도구 준비, 공개 회귀 | 사용자 창 A를 즉시 받을 수 있음 |
| A1 실제 이전 | 20~40분 | 사용자+Codex | 실제 ENEX 최종 저장, 수량·오류·시간·공간 기록 | 1,950/2,272 저장 결과 또는 정확한 실패 로그 |
| A2 실제 모바일 | 15~30분 | 사용자+Codex | Android 연결, 찾기·수정·재접속·첨부 표본·원상복구 | v12 실기기 증거 |
| A3 파일럿 | 같은 날 병렬 30~60분 | 사용자+3명 | 10분 과업을 3명이 병렬 수행 | 최소 3명 결과 |
| A4 최소 수정 | 0~1일 | Codex | P0 즉시, 두 환경에서 재현된 P1만 수정·새 버전·공개 회귀 | P0=0 |
| A5 RC 종료 | 1~2시간 | Codex | SHA·SW·공개 핵심 회귀·한계·롤백·종료 보고 | 5관문 통과 |

외부 일정 지연이 없다면 목표는 **1~3일**이다. 이 값은 약속이 아니라 현재 남은 증거와 한 번의 수정 라운드를 가정한 계획값이다. 파일럿 모집이나 기기 연결이 늦어지면 그만큼 달력시간이 늘어난다.

## 7. 알파에서 제외해 달력시간을 줄이는 항목

다음은 실제 P0/P1로 관찰될 때만 임계 경로에 복귀한다.

- iPhone 교차 확인: Android 1대가 관문을 충족하므로 베타 교차검증
- 10,000노트 벤치마크: 실제 알파 데이터는 1,950노트·766MB이며 이미 미리보기 통과
- 전체 브라우저 전수 매트릭스: 파일럿이 실제 사용하는 브라우저와 Chrome·Edge 핵심만 우선
- 노트북 매핑·태그 정규화·동명이인 고도화
- HEIC·HEIF·M4A 추가 호환성
- Windows 전체 자동 백업 도우미의 실제 계정 종단 자동화
- 클라우드 동기화·협업·수익화·제휴·고급 편집

## 8. 무중단 실행 운영 규칙

- 임계 경로 작업은 WIP 1로 끝까지 밀고, 외부 입력을 기다릴 때만 자율 작업열을 실행한다.
- 사용자 의존 항목을 만나도 전체 실행을 멈추지 않는다. 의존성을 기록하고 다음 안전 작업으로 이동한다.
- 동일 오류는 로그를 근거로 두 번까지 직접 수정·재시도한다. 세 번째에도 같은 차단이면 재현 자료를 남기고 다른 작업으로 이동한다.
- 코드 변경은 구현→정적 검사→관련 회귀→공개 배포→공개 재검증을 한 작업 단위로 끝낸다.
- 사용자에게 필요한 행동은 흩어 묻지 않고 20~40분짜리 한 번의 행동 창으로 묶는다.
- 자동 테스트를 실제 기기·실제 사용자 증거로 바꾸어 말하지 않는다.
- 긴 실행 자체를 목표로 삼지 않는다. 알파 관문을 줄이지 않는 작업은 베타로 이동한다.
